# 2026-04-28

## Debugging: NixOS Server Neovim Startup Errors

### Request
User reported multiple errors on a NixOS server when running `make ts` and opening Neovim interactively.

---

### Issue 1: image.nvim luarocks build failure

**Error:** `image.nvim` repeatedly failed to build because `luarocks`, `python3`, and `lua 5.1` were not available on the NixOS server.

**Investigation:**
- `snacks.lua` had `image = { enable = true }` — a typo (`enable` instead of `enabled`) and a feature that pulls in `image.nvim` as a dependency.
- Even after disabling the feature in opts, `snacks.nvim` still declared `image.nvim` as a dependency in its lazy spec, causing lazy.nvim to attempt a luarocks build.

**Solution (two-part):**
1. `snacks.lua:50` — changed `image = { enable = true }` → `image = { enabled = false }` (also fixes typo).
2. `lazy.lua` — added `rocks = { enabled = false }` to the `require("lazy").setup({})` call to prevent lazy.nvim from attempting luarocks/hererocks builds for any plugin.

---

### Issue 2: `make ts` — attempt to call a table value

**Error:**
```
E5108: Lua: [string ":lua"]:1: attempt to call a table value
```
The `make ts` target was calling `require('nvim-treesitter.install').update({ with_sync = true })()` — an API that no longer exists in the `main` branch rewrite of nvim-treesitter (Neovim 0.12+).

**Investigation:**
- `treesitter.lua` uses the new API: `require("nvim-treesitter").update(nil, ts_install_opts):wait(600000)`.
- The `main` branch has no `TSUpdateSync` command either (confirmed by `E492: Not an editor command` error).
- `lazy = false` on the plugin, so it loads at startup, but the command registration still wasn't available headlessly.

**Solution:**
`Makefile` `ts` target updated to:
```makefile
nvim --headless -c "lua require('nvim-treesitter').update(nil, {summary=true, max_jobs=4}):wait(600000)" -c "qa"
```

---

### Issue 3: obsidian.nvim — "Please specify a valid workspace"

**Error:** `Failed to run config for obsidian.nvim` — `At least... Please specify a valid workspace`.

**Investigation:**
- `obsidian.lua` had no `workspaces` configured in opts.
- A `cond` guard was present but commented out: `-- cond = vim.fn.isdirectory(".obsidian") == 1`.
- On a server with no Obsidian vault, the plugin loaded (via `ft = "markdown"`) and failed during setup.

**Solution:**
Uncommented `cond = vim.fn.isdirectory(".obsidian") == 1` in `obsidian.lua:394` so the plugin only loads when inside an Obsidian vault directory.

---

### Issue 4: Treesitter query error on `:` — Invalid node type "tab"

**Error:**
```
vim/treesitter/query.lua:374: Query error at 113:4. Invalid node type "tab"
```
Appeared every time `:` was pressed (noice.nvim intercepts cmdline).

**Investigation:**
- Neovim 0.12.1's bundled `vim` treesitter queries reference a `tab` node type not present in the installed `vim` parser binary.
- Running `make ts` did not resolve it — the parser update didn't add the `tab` node type.
- `noice.lua` config had no explicit cmdline format override, so noice defaulted to parsing the cmdline as `lang = "vim"` treesitter language.

**Solution:**
Added to `noice.lua` opts:
```lua
cmdline = {
    format = {
        cmdline = { pattern = "^:", icon = "", lang = "" },
    },
},
```
Disables treesitter parsing for the `:` cmdline, eliminating the query error.

### Cross-file Relationships

- **snacks.lua → lazy.lua**: Disabling image in snacks opts wasn't sufficient because lazy.nvim itself handles luarocks builds. Required the `rocks = { enabled = false }` fix in `lazy.lua`.
- **Makefile → treesitter.lua**: The correct headless update command was derived by reading `treesitter.lua:97` which shows the current API: `require("nvim-treesitter").update(nil, ts_install_opts):wait(600000)`.
- **noice.lua → treesitter.lua**: The cmdline treesitter error was noice-triggered but rooted in the `vim` parser. Since `treesitter.lua` uses `vim.treesitter.start()` per buffer (new API, no `highlight.disable` list), the fix had to be applied in noice's config instead.
